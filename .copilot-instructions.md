# Copilot Instructions for Volunteer Exchange Platform

## 1) Project Identity (Always Use These)

- **Plugin name:** Volunteer Exchange Platform
- **Main file:** `volunteer-exchange-platform.php`
- **Namespace root:** `VEP\`
- **Text domain:** `volunteer-exchange-platform`
- **Plugin slug:** `volunteer-exchange-platform`
- **Runtime code:** `src/`

Never introduce legacy names like `ns-vep` or old loader architecture.

## 2) Architecture

- Keep `volunteer-exchange-platform.php` as thin bootstrap.
- Runtime orchestration belongs in `VEP\Plugin\Plugin`.
- Lifecycle hooks are class-based:
  - `VEP\Plugin\Activator::activate`
  - `VEP\Plugin\Deactivator::deactivate`
- Dependency checks belong in `VEP\Plugin\Dependencies::check`.

Do not add old-style classes such as `Includes\Plugin_Loader`.

## 3) Dependency Injection & Composition Root

- Build shared repositories/services once in `src/Plugin/Plugin.php`.
- Inject services into Admin/Ajax/Shortcodes/Frontend classes.
- Keep optional constructor fallback only where it improves compatibility.
- Prefer explicit nullable types: `?Type $dep = null`.
- Do not use implicitly nullable typed params (`Type $dep = null`).

## 4) Service/Repository Boundaries

- Repositories live in `src/Database`.
- Services live in `src/Services` and hold business logic.
- Pages/handlers/shortcodes call services, not raw SQL.
- Reuse service graph where possible; avoid repeated ad-hoc instantiation.

## 5) PHP Standards, Security, and Validation

- Follow WordPress Coding Standards (WPCS).
- For directly loadable PHP files, include guard:
  - `if ( ! defined( 'ABSPATH' ) ) { exit; }`
- Verify permissions for admin actions:
  - `current_user_can( 'manage_options' )`
- Verify nonce on submissions/actions:
  - `wp_verify_nonce`, `check_admin_referer`, `check_ajax_referer`
- Sanitize all input (`sanitize_text_field`, `sanitize_email`, `sanitize_textarea_field`, `absint`, etc.).
- Escape all output (`esc_html`, `esc_attr`, `esc_url`, `wp_kses_post`).

For list-table quick actions:

- Generate nonce-protected URLs (`wp_nonce_url`).
- Verify nonce in handler before service/repository operations.

## 6) File Headers

All PHP files should include header comments.

```php
<?php
/**
 * Short description
 *
 * @package VEP
 * @subpackage <Area>
 */
```

Use `<Area>` by folder responsibility:

- `Plugin`, `Admin`, `Ajax`, `Database`, `Services`, `Shortcodes`, `Frontend`

## 7) Class Structure

- Apply SRP (single responsibility principle).
- Keep related behavior together; avoid “god classes”.
- Use type hints where possible.
- Document public methods with concise PHPDoc.
- For private methods, follow existing local style (do not force underscore naming if file style does not use it).

## 8) Function Documentation (PHPDoc)

- **Public methods:** required PHPDoc.
- **Private/protected methods:** add PHPDoc when behavior is non-trivial.
- Keep docs accurate and update when signatures/behavior change.

Recommended structure:

```php
/**
 * Short description.
 *
 * @param type $arg Description.
 * @return type Description.
 * @throws Exception_Class When relevant.
 * @since 1.0.0
 */
```

Guidance:

- `@param`: include type, name, purpose, and optional/default behavior.
- `@return`: document success and failure outcomes clearly.
- `@throws`: use only if exceptions are actually thrown.
- `@since`: use for public API and significant additions.
- If returning `WP_Error`, state it explicitly.

## 9) Admin UI Patterns

- Keep list rendering in `*ListTable` classes.
- Keep form handling/routing in `*Page` classes.
- Keep admin notices concise and translatable.
- Preserve existing admin routes (`admin.php?page=...`).

## 10) Frontend/JS Patterns

- Frontend logic is in `assets/js/frontend.js` (vanilla JS).
- Scope DOM logic per instance/component (avoid brittle global single-element assumptions).
- Keep both registration modes working (`simple` and `multistep`).
- AJAX responses must return user-friendly localized messages.

## 11) Internationalization (i18n)

- Use text domain: `volunteer-exchange-platform`.
- Keep translations in `/languages`.
- Use helpers by context:
  - `__()` for general strings
  - `esc_html__()` / `esc_attr__()` for escaped output
  - `_n()` for plurals

Example:

```php
__( 'Volunteer Opportunity', 'volunteer-exchange-platform' );
esc_html_e( 'Add New Opportunity', 'volunteer-exchange-platform' );
_n( '%d volunteer', '%d volunteers', $count, 'volunteer-exchange-platform' );
```

When adding or changing user-facing translatable strings:

- Always update `languages/volunteer-exchange-platform.pot` with the new `msgid`.
- Always update `languages/volunteer-exchange-platform-da_DK.po` with corresponding `msgstr` values.
- Always compile the `.po` file to `.mo` in the same task:
  - `php tools/compile-mo.php languages/volunteer-exchange-platform-da_DK.po languages/volunteer-exchange-platform-da_DK.mo`
- Do not leave new translation entries with empty `msgstr` in `volunteer-exchange-platform-da_DK.po`.

## 12) Scope and Change Discipline

- Prefer minimal, targeted changes.
- Follow existing file style and conventions.
- Do not add runtime logic to `tools/`.
- Do not reintroduce outdated architecture/docs.

## 13) Verification

After changes:

- Run `php -l` on changed PHP files.
- Prefer targeted functional checks first (affected page/handler/shortcode).
- If frontend behavior changed, verify the relevant shortcode flows in browser.
